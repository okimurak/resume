### 2019/04 - 2021/06 : インターネット広告系企業

- 職位 : Web エンジニア

#### SSP のクラウド移行

SSP とは、媒体の広告枠販売や広告収益の最大化を支援するためのツールである。2021 年末にシステムが稼働しているデータセンターの廃止が決定したため、オンプレミスで構築されたシステムをクラウドへ移行する必要があり、移行プロジェクトが発足。クラウドインフラへの面白さと魅力を実感し、より深くクラウドインフラと関わるために AWS へ移行するプロジェクトに参画。移行はシステム構成をクラウドネイティブに再設計する形で進行。

##### 役割

- クラウド移行後の技術選定・設計・構築
- CI/CD パイプラインの構築
- プロジェクトのドキュメント整備
- スケジュール・タスク管理

##### 環境

- バックエンド : Java 8, Python3.7
- フロントエンド : JavaScript
- インフラ
  - オンプレミス : CentOS(VM) + MySQL(VM)
  - クラウド
    - アプリケーション実行基盤 : ECS, Lambda, Step Functions
    - DB : Aurora, DynamoDB
    - Storage : S3, EFS
    - Network : VPC, API Gateway, Route53, Fastly
    - ログ・監視 : CloudWatch, Datadog, Mackerel
    - その他 : SES, EventBridge, SSM Parameter Store
- 開発規模は 14 名

##### 実績・ポイント

- 業務アプリケーションのコンテナ化および ECS 移行を推進。その際、ECS が CloudWatch Events の仕様により多重起動してしまう問題が発生。また、処理ごとにリポジトリが分割されていたため、移行が進むにつれリリース担当者の作業負荷と作業時間が増大し、30 分ほどかかっていた。  
そこで Step Functions と DynamoDB を組み合わせた「多重起動防止機構」を組み込んだ ECS 実行基盤を設計・構築。また、各バッチの定義と起動トリガーを一括管理する集約リポジトリを構築した。  
これによりバッチの多重起動問題が完全に解消。リリース時の操作対象が 1 リポジトリに集約されたことで、担当者の負荷が大幅に軽減され、リリース作業時間を 10 分程度まで短縮・安定化させた。  
なお、DB 接続が不要な Web アプリケーションは API Gateway + Lambda、バッチアプリケーションは Lambda に移行した。    

- API Gateway, Lambda, Step Functions　等を用いたサーバーレス構成への移行にあたり、社内および自身の技術知見が不足していることが課題であった。  
公式ドキュメントや他社事例、参加した外部勉強会から得たベストプラクティスを元に、早期から小規模なプロトタイプ作成と検証を反復。設定ミスなどのリスクを最小化するサイクルを確立した。  
技術理解の深化により検証スピードが加速。本番環境への設定ミスをゼロに抑え、スムーズな移行を実現するとともに、得られた知見をプロジェクト全体へ共有しチームの技術力底上げに貢献した。    

- 環境の再現性と透明性を確保するため、Terraform および Serverless Framework を用いたインフラのコード化（IaC）を徹底。さらに GitHub と CircleCI を連携させた CI/CD パイプラインを構築し、GitOps によるビルド・デプロイフローを整備した。  
インフラ設定の「見える化」を実現し、手動運用の排除による一貫性のあるデプロイサイクルを構築した。


#### SSP の商材管理システムの開発・運用

PMP(Private Market Place) で用いる商材情報を管理するためのシステムの追加開発・保守を担当。

##### 役割

- 要件定義、設計、開発、テスト、レビュー、リリース
- スケジュール・タスク管理・オフショアとの調整

##### 環境

- バックエンド : PHP 7.0(FuelPHP)
- フロントエンド : JavaScript(jQuery, Backbone.js), Shell
- インフラ : EC2 + CLB, RDS(MySQL), Lambda, SES, CloudWatch, Mackerel
- 開発規模は 3 名

##### 実績・ポイント

- 追加機能の開発において、要件定義からリリースまでの全工程をリード。開発拠点としてベトナムのオフショアチームを活用する体制であった。  
英語を用いた直接的なコミュニケーションにより、要件の齟齬を排したディレクションを実施。上流工程から一貫して品質管理と進捗管理を担った。  
要件定義からリリースまでを完遂。オフショアチームとの円滑な連携体制を築き、グローバルな開発環境におけるリリースを成功させた。

- 従来、デプロイは手動手順で行われており、デプロイ時に各モジュールのビルドを行っていたため、完了までに 40 分を要していた。また、テストコードが保守されず形骸化しており、品質面でも懸念があった。  
GitHub、CircleCI、CodeDeploy を組み合わせた CI/CD パイプラインを構築し、GitOps による自動デプロイフローを整備。ビルド工程をデプロイ前（CI フェーズ）へ移動させるとともに、テストコードの改修を行い、CI 上での自動テストを組み込んだ。  
デプロイ時間を 40 分から 10 分へと75% 削減し、開発サイクルの高速化を実現した。バグの早期発見が可能になるとともに、これらの知見を社内技術記事として共有し、組織全体の技術力向上に貢献した。

- ログ監視の仕組みがなく、障害発生に気づけない状態であった。また、ログはローカルファイルにのみ出力されていたため、調査のたびに複数ファイルをダウンロードして横断的に確認する必要があり、復旧に多大な時間を要していた。  
Fluentd、CloudWatch Logs、Lambda、Slack を連携させたリアルタイム監視基盤を設計・構築。ログの構造化を目的とした独自のロガーを実装し、CloudWatch Logs Insights で効率的にクエリ検索ができる環境を整備した。  
ログレベルに応じたアラート通知により、ユーザーからの報告前に障害を予見・検知することが可能になった。また、複数ファイルにまたがるログ調査の手間を排除し、障害原因の特定および復旧までのスピードを劇的に向上させた。


### 2015/04 - 2019/03: メーカー系 SIer

- 職務: システムエンジニア

#### 担当プロジェクトと業務

#### 2016/04 - 2019/03 : 官公庁向けプロジェクトのリプレイス

システムエンジニアとして通信指令システムのリプレイスプロジェクトに参画し、外部設計からリリースまでを担当。画面設計、テスト、リリースについては全システム。また、詳細設計、実装については他システムで発生した障害情報を収集する監視機能や、リモートで各サーバや端末を操作する運用管理機能を持った運用監視システムのリーダーを担当。

##### 役割

- システム全体の画面設計
- 運用監視システムリーダー
  - 詳細設計、 開発、テスト、レビュー、リリース
  - スケジュール・タスク管理

##### 環境

- バックエンド : Java 8
- フロントエンド : Java FX
- バッチ : Shell, Python3.2
- インフラ
  - サーバ : RHEL7, Windows Server 2016
  - クライアント : Windows 10 Enterprise
  - DB : Oracle 12c
  - アプリケーションサーバー: WebLogic
- 開発規模は 100 名 (チーム規模 3 名, 10 チーム)

##### 実績・ポイント

- 業務で多種多様な機器を扱うお客様のシステム開発において、要件が非常に高度かつ抽象的な表現が多く、認識の齟齬が発生しやすい状況だった。  
複雑な業務フローを正確にシステムへ落とし込み、多くの機器を扱う現場でも迷わない操作性の高い UI を実現することが求められた。  
まず、業務フローを徹底的に分析し、現状に即した新しいフローを再提案。その上で、UI モックによる画面設計から着手し、段階的に内部ロジックを実装。現場担当者に実際に触れてもらうデモと改修を繰り返す「プロトタイプ型開発」を実践した。  
現場のリアルなフィードバックを反映させることで、操作ステップの削減と視認性の高いレイアウトを実現。お客様との認識齟齬を未然に防ぎ、複雑な機器操作を伴う業務でもミスが起きにくい最適なシステムを構築できた。

- Subversion によるファイルロック運用を行っており、複数人での同時並行開発ができない制限があった。また、ビルド・デプロイ・テストのすべてが手動であり、人的ミスのリスクが高い状態だった。  
そのため、開発リードタイムの短縮と、品質担保のための自動化プロセスの導入が急務だった。  
Git + Jenkins + Coverity による CI/CD 環境の構築を提案・実施した。しかし、Git を知る人がプロジェクトにおらず、この技術習得の壁を打破するために、共通マニュアルの作成、勉強会の開催、3 ヶ月間にわたる個別サポートを粘り強く継続。  
結果、並行開発が可能になったことで開発効率が向上。ビルド・デプロイの自動化により作業ミスを排除し、Coverity による静的解析でバグの早期検知を実現した。導入 3 ヶ月後にはチーム全員が自律的に運用できるレベルまで定着した。

- 運用監視システムのリーダーとしてチーム運営をする中、プロジェクト独自の教育資料が整備されておらず、新人の立ち上がりに平均 1 ヶ月を要した。  
リーダーとしての工数管理と並行し、新人が早期に現場戦力として貢献できる教育体制の構築が必要でした。  
独自の新人オンボーディング資料をゼロから作成した。単なる説明資料ではなく、サンプルコードや実装例を含めた「電卓作成」という実践的なプログラミング課題を盛り込み、手を動かしながら学べる形式にした。  
教育期間を従来の 1 ヶ月から 2 週間へ 50% 短縮することに成功。属人化していた教育プロセスを標準化したことで、教育担当者の負担軽減と新人の早期戦力化を同時に実現した。
